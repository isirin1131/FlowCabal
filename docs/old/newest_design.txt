runner-core/ 重构设计

  核心前提

  CLI-first：Beta 版 99% CLI / 1% UI。每个操作都是原子命令，TUI/GUI 只是调用同样的原子操作的视觉包装。外部 Agent（Claude Code、Cursor、Cline）可以零成本通过 shell 驱动 FlowCabal。

  设计哲学："如果用户不重视生成的东西，那我们也认为它没有价值。"

  7 项职责

  1. Node 管理 — CRUD 节点，nano_id 查询
  2. Target Set（持久化，自动管理）— 新建节点→自动加入，修改节点→自动加入，跑完→自动移除。用户可手动增删。
  3. Stale Roots（持久化，惰性）— 上游节点被修改/重跑时，直接下游加入 stale roots。自身重跑→从 stale roots 移除 + 向自身下游传播。完整的 Possibly Stale 集合按需 BFS 推导，不实时维护。
  4. Todo List（实时计算，不存储）— Target Set + DAG 依赖 → Kahn 拓扑排序 → 分层级列表
  5. Execution — 简单函数：给一组节点（一个 level），并行执行，返回结果。无 RunHandle，无事件总线，无 auto/step 模式。编排逻辑属于调用方。
  6. Version 管理 — 多版本存储，切换当前版本
  7. Preferences — per-node LLM 参数覆盖

  移除的东西

  - 结构性 hash 缓存失效（computePromptHash / computeStructuralHash）— 过时判断完全由 Target Set + Stale Roots 驱动
  - RunHandle / 事件总线（RunEvent、subscribe、advance、abort）— 执行变成简单函数调用
  - RunMode（auto | step）— 不再有执行模式
  - Agent Gates — 冗余，外部 Agent 本身就是 gate
  - Workspace 长会话对象（openWorkspace() → 返回有状态 Workspace 实例）— 替换为无状态原子操作函数
  - agent-inject 缓存失效 — 暂不考虑，用户自行用 git 管理 memory files

  API 模型

  不再是 openWorkspace() → 长生命周期的 Workspace 对象 → 调用方法。

  而是一组无状态原子操作函数，每次调用独立完成：加载→执行→持久化→返回。CLI 每条命令调用一个函数，TUI 内部也是调用同样的函数。

  执行模型

  executeLevel(nodes: NodeDef[], outputs: Map<string, string>, llmConfig: LlmConfig)
    → Promise<Map<string, string>>  // nodeId → output

  给一组同层节点 + 上游 outputs，并行调用 LLM，返回结果。调用方（CLI/TUI/外部 Agent）自行决定逐层推进还是一口气跑完。

  保留的模块

  - workflow.ts — extractNodeDeps()、topoSort()、topoLevels()、computeSubgraph()、validateWorkflow() — 干净的纯函数，Todo List 计算需要
  - resolve.ts — resolveBlocks()、resolveBlocksFull() — 保留（computePromptHash() 可能移除）
  - budget.ts — estimateTokens() — 保留
  - convert.ts — importWorkflow() — 保留
  - cache.ts — readJson/writeJson 保留，workspace 生命周期函数可能重组

对。外部 agent 本身就是 gate——它跑一层、看输出、自己决定要不要继续。把这个逻辑内置到引擎里是重复的。                                                                                       
                                                                                                                                                                                           
那现在 runner-core/ 的设计收敛了：                                                                                                                      
最终职责                                                                                                                                                                                   
                
1. 节点管理 — CRUD nodes，按 id 查询                                                                                                                                                       
2. Target Set — 持久化，自动维护（新建→加入，修改→加入，跑完→移出）
3. Stale Roots — 持久化，懒惰推导 Possibly Stale（上游变化→加入，自身重跑→移出+传播）                                                                                                      
4. Todo List — 实时计算（Target Set + DAG → Kahn 分层）                                                                                                                                    
5. 执行 — 跑一个节点或一层，返回结果，不编排后续                                                                                                                                           
6. 版本管理 — 多版本存储、切换当前版本                                                                                                                                                     
7. Preferences — per-node LLM 参数覆盖                                                                                                                                                     
                                                                                                                                                                                           
执行引擎不再需要 RunHandle、事件总线、auto/step 模式。就是一个函数：给我一组节点，并行跑完，返回结果。编排逻辑留给调用方。

---

现阶段用户可以自行对 memory files 用 git 进行版本管理，将来我们也会原生支持，但现阶段要集中力量把 runner-core 部分做好。所以现在可以不考虑 agent inject 部分的缓存失效，这点是重点。

1. 用户可以添加和编辑节点，可以列出所有节点的 nano_id，可以按照 nano_id 查询节点的信息
2. 用户可以编辑一个 target nodes set，workspace 第一次识别到一个节点时（不管是从 workflow 中导入还是在 workspace 中新创建），这个节点默认在 target nodes set 里面，一个节点若运行结束，它会被自动移出 target nodes set，一个节点若被修改，则自动移入 target nodes set。
2.5. 正如我们不考虑 agent inject 的缓存失效，workspace 的缓存也应当取消，这是基于一点设计哲学：如果用户不重视生成的东西，那我们也认为它没有价值。
3. 基于 1 和 2，用户可以获得一个实时解析的 todo list 和一个实时维护的 “可能过时的节点”，todo list 完全依赖于 target node set 和工作流的依赖结构，它会依照 Kahn 序的解析顺序分层，而 “可能过时的节点” 独立于 target 体系，它只在其上游节点被修改时或被运行时加入，自身运行后则被移出。这个可能过时的节点完全只用来提醒用户这些节点可能过时，用户可以自行决定要不要把他们加入 target nodes set。当然肯定需要一些高效数据结构。
3.5. 第三点是简化设计，第四点则是增加设计。
4. 在 todo list 中，用户将可以为每个节点和每个层设置 agent gate，让 agent 以一些精准的或者模糊的评判标准判断要不要让这个 todo list 继续运行下去。

# Plan to implement

## runner-core/ 重构计划

### Context

当前 `runner-core/` 设计混乱：`state.ts` 过胖（存储+计算+事件全混在一起），`workspace.ts` 是几乎无用的薄包装，`run.ts` 有不必要的 RunHandle/EventBus/auto-step 模式。Beta 版需要 **CLI-first** 的原子操作 API，让外部 Agent（Claude Code、Cursor）零成本驱动。

**核心改变**：从“打开会话对象 → 调用方法”变为“每次调用一个独立函数，加载→执行→持久化→返回”。

### Phase 1: types.ts + schema.ts + paths.ts

#### `types.ts` — 新增
